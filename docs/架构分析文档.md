# 电商微服务系统架构分析文档

## 一、项目整体架构设计方案

### 1.1 架构概述

本项目是一个基于 Spring Cloud 的电商微服务系统，采用领域驱动设计（DDD）思想进行服务划分，覆盖用户、商家、管理员三类角色，并打通订单、售后与 AI 客服闭环。

### 1.2 微服务划分原则

**业务边界原则**：
- 按业务领域划分：用户域、商家域、订单域、支付域、AI域等
- 每个服务负责特定的业务能力，职责单一
- 避免跨服务的强耦合

**数据自治原则**：
- 每个微服务拥有独立的数据库
- 避免跨服务的数据库访问

**技术栈统一原则**：
- 所有服务基于 Spring Boot + Spring Cloud
- 统一使用 Nacos 进行服务注册和配置管理
- 统一的安全认证机制（JWT Token）

### 1.3 服务边界定义

| 服务名称 | 端口 | 核心职责 | 数据库 |
|---------|------|---------|--------|
| ecommerce-gateway | 8080 | API网关、路由转发、CORS处理 | 无 |
| ecommerce-user-service | 8083 | 用户注册登录、权限管理、积分会员 | ecommerce_user |
| ecommerce-merchant-service | 8084 | 商家申请审核、店铺商品管理 | ecommerce_merchant |
| ecommerce-dashboard | 8085 | 商城主页、商品搜索展示 | 无 |
| ecommerce-ai-service | 8082 | AI客服、RAG检索、多Agent路由 | 无 |
| ecommerce-chat-service | 8081 | 聊天会话管理、消息存储 | chat_db |
| ecommerce-cart-service | 8087 | 购物车管理 | cart_db |
| ecommerce-order-service | 8088 | 订单创建、状态管理 | order_db |
| ecommerce-payment-service | 8089 | 支付处理 | payment_db |
| ecommerce-inventory-service | 8086 | 库存管理 | inventory_db |

### 1.4 核心组件

**基础设施层**：
- **Nacos**：服务注册发现 + 配置中心（端口8848）
- **Redis Cluster**：分布式缓存（6节点，端口7000-7005）
- **Qdrant**：向量数据库，支持AI RAG检索（端口6333）
- **RabbitMQ**：消息队列，异步处理（端口5672）
- **MySQL**：关系型数据库，多实例部署

**网关层**：
- **Spring Cloud Gateway**：统一入口，路由规则配置
- CORS跨域处理
- 请求转发到各微服务

## 二、各文件夹的具体含义与功能定位

### 2.1 核心服务文件夹

**ecommerce-gateway/**：
- 作用：API网关，系统的统一入口
- 核心配置：路由规则（routes）、CORS配置
- 路由策略：基于Path的路由，如`/api/user/**`转发到user-service

**ecommerce-user-service/**：
- 作用：用户域核心服务
- 功能模块：
  - `controller/`：账户、认证、管理员、优惠券、会员接口
  - `domain/`：用户账户、优惠券、积分、角色等实体
  - `repository/`：数据访问层（JPA）
  - `service/`：业务逻辑层
  - `security/`：JWT认证、权限控制

**ecommerce-merchant-service/**：
- 作用：商家域核心服务
- 功能模块：商家申请、审核、店铺管理、商品上架
- 特色：集成Redis Cluster和RabbitMQ

**ecommerce-ai-service/**：
- 作用：AI智能客服服务
- 核心技术：Spring AI + MiniMax大模型 + RAG
- 特色功能：多Agent路由、向量检索、图像识别

**ecommerce-chat-service/**：
- 作用：聊天会话管理
- 功能：会话创建、消息存储、状态管理

**ecommerce-cart-service/**、**ecommerce-order-service/**、**ecommerce-payment-service/**、**ecommerce-inventory-service/**：
- 作用：购物车域、订单域、支付域、库存域服务

### 2.2 支持性文件夹

**ecommerce-dashboard/**：
- 作用：商城主页服务
- 功能：商品展示、搜索、推荐

**ecommerce-frontend/**：
- 作用：Vue 3前端应用
- 组件：AiChatDrawer、CartDrawer等

**docker/**：
- 作用：基础设施Docker配置
- 子目录：qdrant、rabbitmq、redis-cluster

**data/uploads/**：
- 作用：文件上传存储
- 子目录：merchant/product、user/avatar

## 三、深度案例分析：ecommerce-chat-service

### 3.1 服务定位

`ecommerce-chat-service` 是一个专门负责聊天会话管理的微服务，主要职责包括：
- 会话的创建、查询、关闭
- 消息的发送、存储、查询
- 未读消息计数
- 售后相关的会话处理

### 3.2 src/目录结构

```
ecommerce-chat-service/src/main/java/org/example/chat/
├── Main.java                    # 启动类
├── client/                      # 外部服务客户端
│   ├── MerchantCatalogClient.java
│   ├── OrderClient.java
│   └── PaymentClient.java
├── controller/                  # REST控制器
│   └── ChatController.java
├── domain/                      # 领域模型
│   ├── ChatConversation.java    # 会话实体
│   ├── ChatMessage.java         # 消息实体
│   ├── ConversationStatus.java  # 会话状态枚举
│   ├── MessageType.java         # 消息类型枚举
│   └── UserRole.java            # 用户角色枚举
├── repository/                  # 数据访问层
│   ├── ChatConversationRepository.java
│   └── ChatMessageRepository.java
├── security/                    # 安全组件
│   ├── AuthenticatedUser.java
│   ├── SecurityConfig.java
│   ├── TokenAuthenticationFilter.java
│   └── TokenService.java
└── service/                     # 业务服务层
    └── ChatConversationService.java
```

### 3.3 各模块职责详解

#### 3.3.1 controller层

**ChatController.java**：
- 职责：暴露REST API接口，处理HTTP请求
- 核心接口：
  - `POST /api/chat/conversations/open` - 打开新会话
  - `POST /api/chat/conversations/open-after-sale` - 打开售后会话
  - `GET /api/chat/conversations` - 获取会话列表
  - `GET /api/chat/conversations/{id}` - 获取会话详情
  - `GET /api/chat/conversations/{id}/messages` - 获取消息列表
  - `POST /api/chat/conversations/{id}/messages` - 发送消息
  - `POST /api/chat/conversations/{id}/read` - 标记已读
  - `POST /api/chat/conversations/{id}/after-sale/action` - 售后操作

#### 3.3.2 domain层

**ChatConversation.java**：
- 职责：会话领域实体，映射数据库表`chat_conversation`
- 核心字段：
  - `storeId`/`storeTitle`：店铺信息
  - `buyerUserId`/`buyerNickname`：买家信息
  - `merchantUserId`/`merchantNickname`：商家信息
  - `bizType`：业务类型（SHOP_SUPPORT/AFTER_SALE）
  - `orderNo`：关联订单号（售后场景）
  - `productId`/`productTitle`：关联商品信息
  - `lastMessageText`/`lastMessageAt`：最后消息信息
  - `buyerUnreadCount`/`merchantUnreadCount`/`adminUnreadCount`：各角色未读计数
  - `status`：会话状态（ACTIVE/CLOSED）

**ChatMessage.java**：
- 职责：消息领域实体，映射数据库表`chat_message`
- 核心字段：
  - `conversationId`：所属会话ID
  - `senderUserId`/`senderRole`/`senderNickname`：发送者信息
  - `clientMsgId`：客户端消息ID（用于去重）
  - `messageType`：消息类型（TEXT/PRODUCT_CARD/AFTER_SALE_ACTION/SYSTEM_NOTICE）
  - `content`：消息内容
  - `payloadJson`：扩展数据JSON

**ConversationStatus.java**：
- 职责：会话状态枚举
- 值：`ACTIVE`（活跃）、`CLOSED`（关闭）

**MessageType.java**：
- 职责：消息类型枚举
- 值：`TEXT`（文本）、`PRODUCT_CARD`（商品卡片）、`AFTER_SALE_ACTION`（售后操作）、`SYSTEM_NOTICE`（系统通知）

#### 3.3.3 repository层

**ChatConversationRepository.java**：
- 职责：会话数据访问接口
- 继承`JpaRepository`，提供CRUD操作
- 自定义查询方法：根据`conversationKey`查询会话

**ChatMessageRepository.java**：
- 职责：消息数据访问接口
- 提供消息的增删查操作

#### 3.3.4 service层

**ChatConversationService.java**：
- 职责：会话业务逻辑核心服务
- 核心方法：
  - `openConversation()` - 打开普通会话
  - `openAfterSaleConversation()` - 打开售后会话
  - `listConversations()` - 获取用户会话列表
  - `getConversation()` - 获取会话详情
  - `listMessages()` - 获取消息列表
  - `sendMessage()` - 发送消息
  - `markRead()` - 标记已读
  - `applyAfterSaleAction()` - 应用售后操作

#### 3.3.5 client层

**MerchantCatalogClient.java**：
- 职责：调用商家商品服务，获取店铺和商品信息

**OrderClient.java**：
- 职责：调用订单服务，获取订单详情

**PaymentClient.java**：
- 职责：调用支付服务，处理退款等操作

#### 3.3.6 security层

**TokenAuthenticationFilter.java**：
- 职责：JWT Token验证过滤器

**TokenService.java**：
- 职责：Token生成与解析服务

**AuthenticatedUser.java**：
- 职责：认证用户信息封装

## 四、微服务间及服务内部通讯机制

### 4.1 微服务间通讯

**服务发现**：
- 使用 Nacos 作为服务注册中心
- 各服务启动时自动注册到 Nacos
- Gateway 通过 `lb://service-name` 进行负载均衡

**HTTP通讯**：
- 使用 Spring 的 `RestClient` 进行HTTP调用
- 通过 `Authorization` 头传递JWT Token

**网关路由配置示例**：
```yaml
routes:
  - id: chat-service
    uri: lb://ecommerce-chat-service
    predicates:
      - Path=/api/chat/**
```

### 4.2 服务内部通讯

**分层调用**：
```
Controller → Service → Repository
Controller → Service → Client（调用外部服务）
```

**依赖注入**：
- 使用 Spring 构造器注入
- 实现松耦合，便于单元测试

**事务管理**：
- 使用 `@Transactional` 注解管理事务
- 确保会话和消息的一致性

### 4.3 接口设计原则

**RESTful设计**：
- 使用标准HTTP动词：GET、POST、PUT、DELETE
- 资源命名清晰：`/api/chat/conversations`、`/api/chat/conversations/{id}/messages`
- 统一响应格式（Map<String, Object>）

**认证机制**：
- JWT Token认证
- 通过Spring Security实现权限控制
- 各服务独立验证Token

**错误处理**：
- 使用 `ResponseStatusException` 返回HTTP状态码
- 统一错误信息格式

## 五、架构设计思想总结

### 5.1 设计思想

**微服务架构**：
- 服务拆分：按业务领域拆分，职责单一
- 独立部署：每个服务可独立开发、部署、扩展
- 技术栈统一：Spring Cloud生态

**领域驱动设计**：
- 领域边界清晰：会话域、消息域分离
- 数据自治：独立数据库
- 业务语言映射：代码结构反映业务概念

**事件驱动架构**：
- 异步处理：使用RabbitMQ进行异步消息传递
- 解耦：服务间通过消息通讯

### 5.2 框架选型依据

**Spring Cloud**：
- 成熟的微服务框架
- 丰富的组件：Gateway、Nacos等
- 社区活跃，文档完善

**Spring Data JPA**：
- 简化数据访问层开发
- 提供Repository模式
- 支持事务管理

**Nacos**：
- 服务注册 + 配置中心一体化
- 支持动态配置更新

### 5.3 模块化组织策略

**按职责分层**：
- Controller层：接口暴露
- Service层：业务逻辑
- Repository层：数据访问
- Client层：外部服务调用

**按功能模块化**：
- 每个功能独立成模块
- 清晰的依赖关系
- 易于测试和维护

**配置外部化**：
- 使用Nacos配置中心
- 环境变量覆盖
- 支持多环境配置

## 六、架构优势与扩展性

### 6.1 架构优势

**高可用性**：
- 服务独立部署，单点故障不影响其他服务
- Redis Cluster提供高可用缓存
- Nacos支持服务健康检查

**可扩展性**：
- 服务可独立水平扩展
- 支持新增服务模块
- 易于集成新功能

**可维护性**：
- 职责清晰，代码结构良好
- 统一技术栈，降低学习成本
- 独立测试，便于定位问题

### 6.2 扩展建议

**消息队列增强**：
- 使用RabbitMQ实现消息异步处理
- 支持消息重试和死信队列

**缓存优化**：
- Redis缓存会话列表和热门会话
- 实现缓存击穿、穿透、雪崩防护

**监控告警**：
- 集成Prometheus+Grafana监控
- 实现服务健康检查和告警

**分布式事务**：
- 使用Seata实现分布式事务
- 保证跨服务数据一致性
