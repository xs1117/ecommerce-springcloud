## 5. 缺陷分析

### 5.1 首页聚合失败

- 现象：`GET /api/home` 返回 500。
- 根因判断：网关到首页聚合链路当前不可用，且 `8081` 端口监听进程不是本项目 Java 服务。
- 业务影响：主页 banner、商品推荐、搜索入口、公告等基础内容无法正常展示。

### 5.2 用户登录失败

- 现象：`POST /api/user/auth/login` 返回 500。
- 根因判断：`8083` 用户服务未处于监听状态，或其后端依赖未满足，导致登录请求无法完成。
- 业务影响：用户无法获得 token，后续所有鉴权页面均无法进入。

### 5.3 服务监听异常

- 现象：`8080`、`8081`、`8082` 的监听进程为 `System(PID 4)`。
- 根因判断：这三个端口当前不对应可确认的本项目 Java 服务实例。
- 业务影响：网关、首页聚合、AI 服务的真实运行状态无法成立，联调测试失去基础。

### 5.4 日志告警

- 日志中存在 `Nacos Config ... is empty`、`Cannot determine local hostname`、`Spring Cloud LoadBalancer ... default cache` 等告警。
- 含义：配置中心内容不完整、主机名解析不稳定、负载均衡缓存采用默认实现。
- 业务影响：这些信息本身不等于功能故障，但会增加联调复杂度，并放大环境不稳定风险。

## 6. 测试结论

结论：不通过。

依据：

- 前端工程可构建、可静态访问，说明前端基础正确。
- 但核心业务接口 `GET /api/home` 和 `POST /api/user/auth/login` 均返回 500。
- 关键端口监听状态与本项目服务不一致，核心后端链路未达到可验收状态。

## 7. 优化建议

1. 先恢复 `8080/8081/8082/8083` 对应的真实 Java 服务监听，再重新执行联调用例。
2. 补全 Nacos 配置，避免 `config ... is empty` 触发默认值回退。
3. 对首页聚合服务和用户服务增加启动自检与健康检查接口，减少“服务已起但链路仍 500”的情况。
4. 在测试阶段优先保留可重复执行的原始证据，包括截图、接口响应、压测输出和日志片段。
5. 补充 `8080 -> 8083/8081/8082` 的路由转发验证，确认网关到下游服务的实际调用链路。
