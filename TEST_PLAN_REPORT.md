# 项目测试计划与阶段性测试报告

- 项目名称：`ecommerce-springcloud`
- 文档版本：`v1.0`
- 编写日期：`2026-05-06`
- 文档状态：`阶段性版本（静态审查已完成，动态执行受环境限制）`
- 适用对象：项目负责人、开发人员、测试人员、课程验收/答辩评审

## 1. 文档目的

本文档用于在“Remotion 视频制作相关代码已从代码库删除”的前提下，给出一份可审计、可落地的测试计划与阶段性测试报告，重点回答以下问题：

1. 删除 Remotion/视频制作链路后，仓库是否仍残留源代码级耦合。
2. 当前电商项目的核心业务功能应如何开展回归测试。
3. 当前仓库的自动化测试覆盖现状如何，是否足以支撑“已验证完成”的结论。
4. 在本轮环境受限的情况下，哪些结果已经被实际验证，哪些仍需后续补测。

为保证文档严谨性，本文将测试结论划分为两类：

- `已执行结果`：已通过仓库检索、配置审查或命令执行获得的事实结论。
- `待验证结果`：尚未在当前环境完整跑通，只能作为计划项或预期项记录，不能表述为“已通过”。

## 2. 项目基线与测试对象

### 2.1 代码库基线

仓库根 `pom.xml` 显示当前项目为多模块 Spring Cloud 电商系统，包含以下后端模块：

- `ecommerce-gateway`
- `ecommerce-user-service`
- `ecommerce-merchant-service`
- `ecommerce-cart-service`
- `ecommerce-inventory-service`
- `ecommerce-order-service`
- `ecommerce-payment-service`
- `ecommerce-ai-service`
- `ecommerce-dashboard`
- `ecommerce-chat-service`

同时包含前端模块：

- `ecommerce-frontend`（Vue 3 + Vite）

### 2.2 Remotion 删除后的回归重点

本轮测试的核心不是验证视频产物质量，而是验证“删除视频制作相关实现后”：

- 前端构建配置未残留 Remotion/Vite 插件依赖。
- 后端模块未残留视频生成、渲染或 PPT 生成调用链。
- 现有核心电商主流程未因删除无关能力而受影响。

### 2.3 当前自动化测试现状

已识别到的自动化测试极少，主要集中在 AI 服务：

- `ecommerce-ai-service/src/test/java/org/example/rag/KnowledgeChunkerTest.java`
- `ecommerce-ai-service/src/test/java/org/example/service/AiCustomerServiceActionTest.java`
- `ecommerce-ai-service/src/test/java/org/example/service/ConfirmationServiceTest.java`
- `ecommerce-ai-service/src/test/java/org/example/service/RagServiceTest.java`

截至本次审查：

- 其余后端模块未发现成体系的 `src/test` 用例。
- 前端 `package.json` 未定义 `test` 脚本，仅包含 `dev`、`build`、`preview`。

这意味着当前仓库**缺乏覆盖核心业务的自动化回归网**，属于本次报告的重要风险结论。

## 3. 测试范围

### 3.1 本次纳入范围

1. Remotion/视频制作残留引用排查。
2. 前端工程配置与构建链路回归。
3. 网关路由与主要微服务接口面回归规划。
4. 用户、商家、购物车、库存、订单、支付、聊天、AI、首页推荐等核心业务链路。
5. 当前自动化测试现状盘点与风险评估。

### 3.2 本次不纳入范围

1. PPT 文件内容质量评审。
2. 性能压测、并发压测、容量评估。
3. 安全渗透、代码审计、合规审计。
4. 第三方中间件自身正确性验证（Nacos/Redis/Qdrant/MySQL 产品级验证）。
5. 生产部署验证。

## 4. 测试目标

### 4.1 总体目标

- 证明代码库中不存在明显的 Remotion/视频制作残留源码耦合。
- 建立一份可执行的测试计划，覆盖当前真实业务功能面。
- 给出当前仓库是否具备“已完成回归验证”条件的客观判断。

### 4.2 质量目标

- `P0/P1` 级缺陷在上线前必须清零。
- 核心链路必须至少通过一次可重复的构建/测试/手工回归。
- 删除 Remotion 后，前后端构建链路不应因缺失依赖报错。
- 核心业务流应具备最小自动化保障，而不是完全依赖人工检查。

## 5. 测试策略

### 5.1 分层策略

1. `静态检查`
   - 检索 `remotion`、`@remotion`、视频制作关键字、构建配置残留。
   - 审查前端 `package.json`、`vite.config.js`、后端 `pom.xml` 与关键模块结构。

2. `单元测试`
   - 优先验证 AI 服务中与售后动作、知识切片、确认令牌相关逻辑。
   - 补齐用户鉴权、订单状态机、购物车金额计算、库存锁定/释放等纯逻辑模块测试。

3. `接口集成测试`
   - 验证控制器层输入校验、状态码、异常处理、鉴权边界。
   - 覆盖用户、商家、订单、支付、聊天、AI 等关键 API。

4. `跨服务联调测试`
   - 验证订单支付、库存扣减、售后申请、AI 介入、聊天消息联动。
   - 验证网关前缀转发是否正确。

5. `前端冒烟/E2E 测试`
   - 验证登录、首页、搜索、商品详情、购物车、下单、订单、商家中心、管理员审核、聊天等主路径。

### 5.2 覆盖优先级

- `高优先级`：登录鉴权、首页、商品检索、购物车、下单、支付、订单查询、售后、聊天、AI 售后辅助。
- `中优先级`：积分商城、公告管理、评论管理、商家上传。
- `低优先级`：样式细节、非关键提示文案、展示型页面的视觉微差异。

## 6. 准入准出标准

### 6.1 准入标准

开始执行完整回归前，应满足：

1. 所有模块代码冻结到明确版本。
2. 本地或 CI 环境可正常运行 Java、Maven、Node、Vite。
3. MySQL、Nacos、Redis、Qdrant 等基础依赖可启动。
4. 网关及各微服务配置可用。

### 6.2 准出标准

宣布“回归完成”前，应满足：

1. 无 `P0/P1` 级遗留缺陷。
2. 前端构建成功。
3. 关键后端模块至少完成一次自动化测试或接口联调。
4. 删除 Remotion 后的静态残留检查通过。
5. 核心业务链路通过率达到 `100%`。

### 6.3 当前状态判断

截至本报告编写时，**准出标准尚未满足**。原因不是已证实存在业务缺陷，而是：

- 动态验证未完整跑通。
- 核心自动化覆盖明显不足。
- 当前宿主机环境存在内存/分页文件限制，阻塞了构建与测试执行。

- Nacos

### 7.3 已识别环境风险

当前机器存在明显资源问题：

- Java 进程在执行 Maven 时出现本机分页文件/内存不足。
- Vite 构建依赖 `esbuild`，在当前环境下出现子进程启动受限与内存不足问题。

因此，本轮报告必须区分“代码问题”和“环境问题”，不能将资源故障误判为功能故障。

## 10. 阶段性问题与风险

### 10.1 高风险问题

1. **动态回归未闭环**
   - 当前只能确认“静态上未发现 Remotion 残留”，不能确认“所有运行时链路均无回归”。

2. **自动化测试覆盖严重不足**
   - 绝大多数核心服务无单元/集成测试。
   - 一旦后续继续重构，缺乏足够的回归保护网。

3. **测试环境不可重复**
   - 宿主机分页文件/可用内存不足，导致 Java/Node 构建链无法稳定运行。
   - 若不解决环境问题，后续任何“已通过”结论都缺乏可信度。

### 10.2 中风险问题

1. **跨服务联动尚未实测**
   - 订单、支付、库存、聊天、AI 售后之间的状态一致性尚未验证。

2. **前端无自动化测试入口**
   - 当前前端仅具备构建脚本，不具备 Vitest/Playwright 等自动化回归能力。

## 11. 阶段性质量结论

### 11.1 已确认结论

1. 代码库中**未发现明显的 Remotion/视频制作源码残留引用**。
2. 前端依赖与 `vite.config.js` **未发现 Remotion 相关依赖或插件残留**。
3. 当前项目的主要运行形态仍是**电商微服务 + Vue 前端**，而非视频制作系统。

### 11.2 不能下结论的部分

以下结论目前**不能**严谨地下达：

- “前端构建已经通过”
- “所有后端模块测试已经通过”
- “删除 Remotion 后系统运行完全正常”

原因不是已经发现业务错误，而是**当前环境阻塞了完整验证**。

### 11.3 综合判定

本轮阶段性判定如下：

- `静态回归结论`：**通过**
- `动态回归结论`：**未验证完成**
- `发布/验收建议`：**暂不建议以“已完整回归验证”状态对外确认**

更严格地说，当前最准确的表述应为：

> 基于仓库静态审查，未发现 Remotion 删除后的明显残留耦合；但由于测试环境资源限制及自动化覆盖不足，动态回归尚未闭环，项目仍需补测后方可形成最终验收结论。

## 12. 后续补测与整改建议

### 12.1 优先补测顺序

1. 修复环境问题
   - 扩大 Windows 分页文件。
   - 保证 Maven 本地仓库可访问。
   - 保证 Node/esbuild 可正常派生子进程。

2. 补跑基础验证
   - `ecommerce-ai-service` 单元测试。
   - 前端 `vite build`。
   - 网关 + 用户服务 + 商家服务最小联调。

3. 补齐自动化测试
   - 用户鉴权测试。
   - 商家申请与审核测试。
   - 订单/支付/库存状态联动测试。
   - 聊天/售后/AI 辅助动作测试。
   - 前端登录、下单、订单查询冒烟测试。

### 12.2 推荐的自动化建设目标

- 单元测试：覆盖核心领域逻辑与状态机。
- 集成测试：覆盖控制器、服务层与数据库/Feign 边界。
- E2E 测试：覆盖登录、搜索、加购、下单、支付、售后、管理员审核等主链路。
- CI：在每次提交或合并请求时自动执行构建与关键测试。
